# Task 20 결재 프로세스 안정화 설계

작성일: 2026-06-05, Asia/Seoul

## 목표

외부 체험과 Oracle VM 배포 전에 더이음 전자결재 앱의 핵심 결재 프로세스가 상태, 권한, 첨부파일, 알림, 삭제 정책에서 일관되게 동작하도록 회귀 안전망을 강화한다.

## 범위

Task 20은 새 업무 기능을 추가하지 않는다. 기존 동작을 기준으로 프로세스 경계 조건을 테스트하고, 테스트가 드러내는 결함만 최소 수정한다.

포함 범위:

- 신청서 상태 전이: 임시저장, 취소, 제출, 결재중, 승인완료, 반려
- 결재 순서: 현재 결재자만 처리 가능, 미래 결재자는 차례가 오기 전 처리 불가
- 권한 읽기: 작성자, 현재 결재자, 처리 완료 결재자, 관리자 권한 확인
- 첨부파일 접근: 신청서 상세 읽기 권한과 동일한 기준으로 파일 content 접근 제한
- 알림 안정성: 알림은 수신자만 조회하고 읽음 처리 가능
- 삭제 제한: 임시저장/취소 외 상태는 작성자와 관리자 모두 완전 삭제 불가

제외 범위:

- 체험자 30명 seed 생성
- 첨부파일 용량/개수/보존 정책의 운영 문서화와 구현
- Oracle VM 생성 또는 외부 URL 배포
- 공휴일, 반복 일정, 외부 캘린더 연동

## 접근 방식

백엔드는 이미 서비스 테스트와 MockMvc API 테스트가 있으므로, 새 안정화 검증도 같은 위치에 추가한다. 프로덕션 코드 변경은 테스트가 실제 결함을 보여줄 때만 수행한다.

주요 테스트 파일:

- `backend/src/test/java/com/theieum/approval/application/ApplicationSubmissionTest.java`
- `backend/src/test/java/com/theieum/approval/application/ApprovalActionTest.java`
- `backend/src/test/java/com/theieum/approval/api/ApiAuthorizationTest.java`

## 성공 기준

- 신청자 아닌 사용자의 제출/첨부/수정/취소가 차단된다.
- 취소 신청서는 바로 제출할 수 없고, 재수정으로 다시 `DRAFT`가 된 후에만 제출 가능하다.
- 다단계 결재에서 미래 결재자는 차례가 오기 전 상세/첨부/승인 처리가 불가하다.
- 현재 결재자가 처리한 뒤 다음 결재자에게만 처리 가능 상태가 이동한다.
- 승인완료와 반려 신청서는 API와 서비스 모두 완전 삭제가 차단된다.
- 알림 읽음 처리는 알림 수신자만 수행할 수 있다.

- 전체 확인: 관련 백엔드 targeted tests, `git diff --check`, 가능하면 프론트 테스트와 staging smoke를 실행한다.
