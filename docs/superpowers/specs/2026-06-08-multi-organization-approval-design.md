# 다중 소속과 겸직 결재 기준 조직 설계

작성일: 2026-06-08, Asia/Seoul

## 목표

사용자가 1개 이상의 조직에 소속될 수 있게 하고, 겸직자는 신청서 작성 시 본인의 소속 중 결재 기준 조직을 선택할 수 있게 한다. 선택된 조직은 예상 결재선과 실제 제출 결재선 산정에 동일하게 사용하며, 제출 후에는 기존처럼 결재 단계 스냅샷을 고정한다.

## 범위

- `user_organizations` 테이블을 추가해 사용자별 메인 소속과 겸직 소속을 모두 저장한다.
- 기존 `users.organization_id`는 대표 소속 미러 값으로 유지한다.
- 관리자 사용자 관리 화면에서 `users.organization_id` 직접 입력을 막고, 사용자 소속 목록에서 대표 소속을 지정하게 한다.
- 대표 소속 변경 시 `user_organizations.primary_flag`와 `users.organization_id`를 같은 트랜잭션에서 동기화한다.
- 신청서 작성 화면에서 본인의 활성 소속 목록 중 결재 기준 조직을 선택한다.
- 예상 결재선 API와 제출 API는 선택된 결재 기준 조직이 신청자의 활성 소속인지 검증한다.
- 결재선 산정 결과에서 같은 결재자가 두 번 이상 나오면 먼저 나온 단계만 유지한다.
- 기존 신청서는 마이그레이션 시 당시 신청자의 `users.organization_id`를 `applications.approval_organization_id`로 채운다.
- 조직 seed에 더이음사랑의교회 조직 트리를 추가한다.

## 비범위

- 사람 이름 기반 실제 교회 구성원 seed 등록은 포함하지 않는다.
- 부서별 역할 권한 분리는 포함하지 않는다.
- 병렬결재, 대리결재, 전결, 금액 조건 결재선은 포함하지 않는다.
- 제출 후 진행 중인 신청서의 결재 기준 조직 변경은 포함하지 않는다.
- `users.organization_id` 컬럼 제거는 포함하지 않는다.

## 현재 구조

현재 사용자는 `users.organization_id` 하나만 가진다. 결재선 산정은 신청자의 단일 조직을 기준으로 `APPLICANT_ORG`, `PARENT_ORG`, `ROOT_ORG`를 해석한다. 관리자 사용자 관리 화면도 단일 조직 선택을 전제로 한다.

이 구조에서는 한 사람이 예배부와 총무부를 겸직하거나, 부와 팀 중 어느 조직 기준으로 결재를 올릴지 선택해야 하는 경우를 표현하기 어렵다.

## 조직 트리

다락방은 이번 기준정보에서 제외한다. 조직은 기존 `organizations` 트리 구조를 사용한다.

```text
더이음사랑의교회
├─ 예배부
│  ├─ 찬양팀
│  ├─ 미디어팀
│  ├─ 새가족팀
│  └─ 중보기도팀
├─ 총무부
│  ├─ 기획팀
│  └─ 시설팀
├─ 재정부
│  ├─ 회계팀
│  └─ 감사팀
└─ 미래준비부
   ├─ 이음씨드
   └─ 이음키즈
```

레벨은 다음처럼 해석한다.

- 레벨 1: 최상위 조직
- 레벨 2: 부
- 레벨 3: 팀

## 데이터 모델

### user_organizations

사용자 소속의 정식 원본 테이블이다.

- `id`
- `user_id`
- `organization_id`
- `primary_flag`
- `sort_order`
- `active`
- `created_at`

제약은 다음과 같이 둔다.

- 같은 사용자와 조직 조합은 중복 등록할 수 없다.
- 활성 사용자마다 활성 대표 소속은 정확히 1개여야 한다.
- 대표 소속은 반드시 활성 소속이어야 한다.
- 대표 소속 row의 `organization_id`는 `users.organization_id`와 일치해야 한다.

PostgreSQL에서는 `where primary_flag = true and active = true` 조건의 partial unique index로 사용자당 활성 대표 소속 1개를 보장한다. 애플리케이션 서비스는 대표 소속 변경을 한 트랜잭션에서 처리해 `users.organization_id` 미러 값을 함께 갱신한다.

### users.organization_id

기존 화면, 인증 응답, 목록 조회, 기존 코드 호환을 위한 대표 소속 미러 값으로 유지한다. 새 기능에서 소속 목록의 원본은 `user_organizations`이며, `users.organization_id`를 직접 편집하는 API와 UI는 제공하지 않는다.

### applications.approval_organization_id

신청자가 제출 시 선택한 결재 기준 조직을 저장한다.

- 임시저장과 수정 중에는 선택값을 저장할 수 있다.
- 제출 시에는 선택 조직이 신청자의 활성 소속인지 다시 검증한다.
- 제출 이후에는 결재선 스냅샷과 함께 고정한다.

기존 신청서는 마이그레이션 시 신청자의 현재 `users.organization_id`를 기준으로 채운다.

## 마이그레이션

마이그레이션은 기존 사용자의 단일 조직을 대표 소속으로 자동 이관한다.

1. `user_organizations` 테이블을 생성한다.
2. 모든 기존 사용자에 대해 `users.organization_id`를 `primary_flag = true`, `active = true` 소속으로 삽입한다.
3. `applications.approval_organization_id` 컬럼을 추가한다.
4. 기존 신청서의 `approval_organization_id`를 신청자의 현재 `users.organization_id`로 채운다.
5. 새 조직 트리를 seed에 추가한다.

기존 seed 사용자의 ID와 기존 핵심 결재선 동작은 유지한다. 새 교회 조직 트리는 사람 이름 배정 없이 기준정보로 제공한다.

## 결재 기준 조직 선택

신청서 작성 화면은 로그인 사용자의 활성 소속 목록을 조회해 결재 기준 조직 선택지를 제공한다.

- 활성 소속이 1개면 기본 선택한다.
- 활성 소속이 2개 이상이면 대표 소속을 기본 선택하되 사용자가 바꿀 수 있다.
- 선택지가 없는 사용자는 신청서를 제출할 수 없다.
- 선택 조직이 본인의 활성 소속이 아니면 예상 결재선과 제출 모두 차단한다.

예상 결재선 API는 결재 유형과 결재 기준 조직을 입력으로 받아 같은 검증과 같은 산정 로직을 사용한다. 제출 API는 별도 조직 ID를 받지 않고, 임시저장/수정에 저장된 `approvalOrganizationId`를 제출 시점에 다시 검증한 뒤 결재선을 생성한다.

## 결재선 산정 규칙

선택된 결재 기준 조직을 신청자 조직으로 보고 기존 스코프를 해석한다.

선택 조직이 팀이면 다음처럼 해석한다.

- `APPLICANT_ORG`: 선택한 팀
- `PARENT_ORG`: 상위 부
- `ROOT_ORG`: 최상위 조직

선택 조직이 부이면 다음처럼 해석한다.

- `APPLICANT_ORG`: 선택한 부
- `PARENT_ORG`: 최상위 조직
- `ROOT_ORG`: 최상위 조직

`PARENT_ORG`와 `ROOT_ORG`가 같은 조직으로 해석되거나, 다른 조직 스코프에서 결과적으로 같은 결재자가 나오면 먼저 나온 결재자만 유지한다. 같은 결재자가 뒤 단계에 다시 나오면 그 단계는 생략한다. 이 정책은 한 신청서에서 같은 사람이 반복 결재하는 상황을 막기 위한 안전장치다.

결재선 산정 결과가 중복 제거 후 비어 있으면 기존처럼 제출과 미리보기를 실패 처리한다.

## 관리자 화면

사용자 관리 화면은 기존 단일 `조직` 입력을 소속 관리 섹션으로 대체한다.

- 사용자 목록에는 대표 소속을 우선 표시한다.
- 사용자 상세/편집 폼에는 소속 조직 목록을 표시한다.
- 소속 row는 조직, 대표 여부, 활성 여부를 가진다.
- 대표 여부는 radio처럼 한 사용자당 하나만 선택할 수 있게 한다.
- 저장 시 대표 소속이 없거나 여러 개면 저장을 막는다.
- 저장 시 같은 조직이 중복되면 저장을 막는다.
- 대표 소속 변경은 `users.organization_id`까지 함께 반영한다.

기존 관리자 API 응답의 `organization_id`, `organization_name`은 대표 소속 기준으로 유지한다. 새 응답에는 `organizationMemberships` 필드로 소속 목록을 함께 내려준다.

## API 방향

관리자 API는 사용자 소속 목록을 포함하도록 확장한다.

```text
GET /api/admin/users
POST /api/admin/users
PUT /api/admin/users/{id}
```

신청자 API는 현재 사용자의 선택 가능한 결재 기준 조직 목록을 제공한다.

```text
GET /api/applications/approval-organizations
```

예상 결재선 API는 결재 기준 조직을 쿼리 파라미터로 받는다.

```text
GET /api/applications/approval-preview?approvalTypeId=1&approvalOrganizationId=10
```

신청서 생성/수정 요청은 `approvalOrganizationId`를 포함한다. 임시저장 생성과 수정 시 선택된 결재 기준 조직을 신청서에 저장하고, 제출 API는 신청서에 저장된 `approvalOrganizationId`만 사용한다. 제출 API에서 별도 조직 ID를 받아 즉시 바꾸는 흐름은 제공하지 않는다.

## 테스트

- 마이그레이션 테스트: 기존 사용자마다 대표 소속 row가 1개 생성된다.
- 마이그레이션 테스트: 기존 신청서의 `approval_organization_id`가 신청자 대표 소속으로 채워진다.
- 서비스 테스트: 대표 소속 변경 시 `user_organizations.primary_flag`와 `users.organization_id`가 동기화된다.
- 서비스 테스트: 대표 소속이 없거나 여러 개면 사용자 저장이 실패한다.
- 서비스 테스트: 겸직 사용자는 본인의 여러 활성 소속 중 하나를 결재 기준 조직으로 선택할 수 있다.
- 서비스 테스트: 본인 소속이 아닌 조직으로 예상 결재선 조회나 제출을 시도하면 실패한다.
- 결재선 테스트: 팀 선택 시 `APPLICANT_ORG`, `PARENT_ORG`, `ROOT_ORG`가 팀, 부, 최상위로 해석된다.
- 결재선 테스트: 부 선택 시 `PARENT_ORG`와 `ROOT_ORG`가 최상위로 해석되고 같은 결재자는 중복 생성되지 않는다.
- 프론트 테스트: 사용자 관리에서 소속 목록과 대표 소속을 편집할 수 있다.
- 프론트 테스트: 신청서 작성 화면에서 활성 소속 목록을 불러오고 대표 소속을 기본 선택한다.
- 프론트 테스트: 선택한 결재 기준 조직으로 예상 결재선을 다시 불러온다.

## 결정 사항

- A안으로 진행한다. `users.organization_id`는 대표 소속 미러로 유지하고, `user_organizations`를 소속 원본으로 사용한다.
- `user_organizations`에는 메인 소속과 겸직 소속을 모두 저장한다.
- 관리자 화면에서 `users.organization_id` 직접 수정은 제공하지 않는다.
- 신청자는 본인의 모든 활성 소속 중 결재 기준 조직을 선택할 수 있다.
- 조직 스코프 해석은 레벨 규칙을 따른다. 부에서 `PARENT_ORG`를 보면 최상위 조직으로 올라간다.
- 같은 결재자가 중복으로 산정되면 먼저 나온 단계만 유지한다.
- 교회 사람 이름 seed는 이번 범위에 포함하지 않고, 조직 트리만 추가한다.
