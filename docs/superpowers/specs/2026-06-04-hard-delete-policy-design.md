## 1. 목적

현재 관리 화면의 삭제 동작은 대부분 데이터를 실제로 지우지 않고 `active=false`로 전환하는 비활성 처리이다. 이 방식은 운영 중 데이터 보존과 복구 가능성 측면에서 유용하지만, 테스트 데이터나 잘못 생성한 기준정보를 완전히 정리하기 어렵다.

Task 16의 목적은 기존 비활성 기능을 유지하면서, 관리자와 사용자가 필요한 범위에서 데이터를 완전히 삭제할 수 있는 정책과 UX를 추가하는 것이다. 단, 결재가 진행되었거나 업무 기록으로 남아야 하는 데이터는 삭제하지 않는다.

## 3. 삭제 유형

### 비활성화

기존 삭제 동작은 운영상 안전한 "비활성화"로 유지한다. 사용자는 목록에서 비활성 데이터를 계속 확인할 수 있고, 필요한 경우 수정 화면에서 다시 활성화할 수 있다.

대상:

- 사용자
- 조직
- 직위
- 결재선
- 조직별 예외 결재자

### 완전 삭제

완전 삭제는 DB row를 제거하고 복구할 수 없는 위험 작업이다. 버튼명과 확인 문구에서 비활성화와 명확히 구분한다.

대상:

- 사용자
- 조직
- 직위
- 결재선
- 조직별 예외 결재자
- `DRAFT` 또는 `CANCELED` 상태의 신청서

## 4. 권한 정책

### 기준정보

기준정보 완전 삭제 권한은 `ADMIN`에게만 부여한다.

- `ADMIN`: 사용자, 조직, 직위, 결재선, 예외 결재자 완전 삭제 가능
- `MANAGER`: 기존 관리 조회와 비활성화 흐름은 유지하되 완전 삭제 불가
- `APPROVER`, `APPLICANT`: 기준정보 완전 삭제 불가

조직, 직위, 결재선, 예외 결재자 관리는 현재 `MANAGER`도 일부 수행할 수 있지만, 물리 삭제는 복구 불가 작업이므로 `ADMIN`으로 제한한다.

### 신청서

신청서 완전 삭제 권한은 작성자 본인과 `ADMIN`에게만 부여한다.

- 작성자 본인: 본인의 `DRAFT`, `CANCELED` 신청서만 완전 삭제 가능
- `ADMIN`: 모든 사용자의 `DRAFT`, `CANCELED` 신청서 완전 삭제 가능
- `MANAGER`: 신청서 완전 삭제 불가
- `APPROVER`: 신청서 완전 삭제 불가

신청서 상태가 `IN_APPROVAL`, `APPROVED`, `REJECTED`이면 누구도 완전 삭제할 수 없다.

## 5. 엔티티별 삭제 조건

### 사용자

사용자는 다음 데이터에서 참조되지 않을 때만 완전 삭제할 수 있다.

- 신청서의 신청자
- 결재 단계의 원 결재자
- 결재 이력의 원 결재자 또는 처리자
- 첨부파일 업로더
- 알림 수신자
- 결재선 단계의 직접 결재자
- 조직별 예외 결재자

참조가 하나라도 있으면 완전 삭제를 차단하고, "신청서 또는 결재 이력에서 사용 중인 사용자입니다. 비활성화만 가능합니다."처럼 운영자가 이해할 수 있는 메시지를 반환한다.

### 조직

조직은 다음 조건을 모두 만족할 때만 완전 삭제할 수 있다.

- 하위 조직이 없다.
- 소속 사용자가 없다.
- 조직별 예외 결재자 설정에서 참조되지 않는다.

상위/하위 조직 구조가 있는 상태에서 부모 조직을 먼저 삭제하면 조직 트리가 깨질 수 있으므로, 하위 조직을 먼저 정리하게 한다.

### 직위

직위는 다음 조건을 모두 만족할 때만 완전 삭제할 수 있다.

- 해당 직위를 가진 사용자가 없다.
- 결재선 단계에서 직위 기준 결재자로 참조되지 않는다.

### 결재선

결재선은 신청서 생성 이후 업무 기록과 직접 연결되지 않는다. 따라서 결재선 완전 삭제 시 하위 `approval_line_steps`를 먼저 삭제하고, 이후 `approval_lines` row를 삭제한다.

다만 삭제 대상 결재선이 active 상태라면 확인 모달에서 "현재 활성 결재선입니다. 삭제하면 새 신청서에서 사용할 수 없습니다."를 명확히 보여준다.

### 조직별 예외 결재자

조직별 예외 결재자는 독립 설정 데이터로 보고 완전 삭제를 허용한다. 삭제 전에는 대상 결재 유형, 조직, 예외 결재자 이름을 확인 모달에 표시한다.

### 신청서

신청서는 `DRAFT` 또는 `CANCELED` 상태에서만 완전 삭제할 수 있다.

삭제 시 함께 제거할 데이터:

- `attachments`
- 실제 첨부파일
- `application_approval_steps`
- `notification_events` 중 해당 신청서를 참조하는 이벤트
- `applications`

`DRAFT`와 `CANCELED` 신청서는 결재 이력이 없어야 정상이다. 만약 결재 이력이 존재하면 데이터 정합성 보호를 위해 삭제를 차단한다.

## 6. API 설계

기존 `DELETE` API는 비활성화 의미로 유지한다. 완전 삭제는 별도 경로를 사용한다.

기준정보:

- `DELETE /api/admin/users/{id}/hard-delete`
- `DELETE /api/admin/organizations/{id}/hard-delete`
- `DELETE /api/admin/positions/{id}/hard-delete`
- `DELETE /api/admin/approval-lines/{id}/hard-delete`
- `DELETE /api/admin/approval-org-exceptions/{id}/hard-delete`

신청서:

- `DELETE /api/applications/{id}/hard-delete`
- `DELETE /api/admin/applications/{id}/hard-delete`

응답:

- 성공: `204 No Content`
- 권한 없음: `403 Forbidden`
- 대상 없음: `404 Not Found`
- 상태 또는 참조 때문에 삭제 불가: `409 Conflict`

참조 때문에 삭제할 수 없는 경우에는 프론트가 그대로 표시할 수 있는 한국어 메시지를 내려준다.

## 7. 프론트 UX

### 관리 기준정보 화면

기존 삭제 버튼은 "비활성화"로 명확히 표시한다. `ADMIN`에게만 "완전 삭제" 버튼을 추가한다.

완전 삭제 버튼은 다음 원칙을 따른다.

- 비활성화 버튼보다 시각적으로 더 위험한 액션으로 표시한다.
- 클릭 시 즉시 삭제하지 않고 확인 모달을 연다.
- 모달 제목은 "완전 삭제"로 한다.
- 모달 본문에 "복구할 수 없습니다."를 포함한다.
- 대상 이름과 ID를 표시한다.
- 서버에서 삭제를 차단하면 모달을 닫지 않고 오류 메시지를 보여준다.

### 신청서 상세 화면

신청서 상세 화면에는 다음 조건에서만 완전 삭제 버튼을 보여준다.

- 상태가 `DRAFT` 또는 `CANCELED`
- 현재 사용자가 작성자 본인이거나 `ADMIN`

작성자 본인에게는 "신청서 삭제"로 표시하고, 관리자 화면에서는 "완전 삭제"로 표시한다. 두 경우 모두 확인 모달에서 복구 불가와 첨부파일 삭제를 안내한다.

## 8. 데이터 흐름

1. 사용자가 완전 삭제 버튼을 누른다.
2. 프론트가 확인 모달을 연다.
3. 사용자가 확인한다.
4. 프론트가 hard-delete API를 호출한다.
5. 백엔드가 권한을 확인한다.
6. 백엔드가 상태와 참조 여부를 확인한다.
7. 삭제 가능하면 관련 하위 데이터를 먼저 삭제한다.
8. 신청서 첨부파일이 있으면 실제 파일을 삭제한다.
9. 최종 대상 row를 삭제한다.
10. 프론트가 목록을 다시 불러오고 성공 메시지를 표시한다.

실제 파일 삭제가 실패하면 DB 삭제도 롤백되도록 처리한다. 로컬 파일 저장소 기준으로 파일이 이미 없는 경우에는 삭제 성공으로 간주할 수 있다.

## 9. 에러 처리

완전 삭제 실패는 사용자가 다음 행동을 결정할 수 있게 구체적으로 안내한다.

- 참조 중인 사용자: 비활성화 사용 안내
- 하위 조직 존재: 하위 조직 먼저 정리 안내
- 소속 사용자 존재: 사용자 이동 또는 비활성화 안내
- 결재 이력 존재: 업무 기록 보호로 삭제 불가 안내
- 신청서 상태 불가: 임시저장 또는 취소 상태만 삭제 가능 안내
- 첨부파일 삭제 실패: 파일 저장소 상태 확인 안내

서버 로그에는 대상 엔티티, ID, 요청자 ID, 실패 사유를 남긴다.

## 10. 테스트 계획

### 백엔드

- `ADMIN`은 참조 없는 사용자/조직/직위/결재선/예외 결재자를 완전 삭제할 수 있다.
- `MANAGER`는 기준정보 완전 삭제 API에서 `403`을 받는다.
- 참조 중인 사용자/조직/직위는 `409`로 차단된다.
- 결재선 완전 삭제 시 하위 단계가 함께 삭제된다.
- 신청자 본인은 본인의 `DRAFT`, `CANCELED` 신청서를 완전 삭제할 수 있다.
- `ADMIN`은 다른 사용자의 `DRAFT`, `CANCELED` 신청서를 완전 삭제할 수 있다.
- `MANAGER`와 `APPROVER`는 신청서를 완전 삭제할 수 없다.
- `IN_APPROVAL`, `APPROVED`, `REJECTED` 신청서는 `ADMIN`도 완전 삭제할 수 없다.
- 신청서 삭제 시 첨부 DB row와 실제 파일이 함께 삭제된다.

### 프론트엔드

- `ADMIN`에게만 기준정보 완전 삭제 버튼이 보인다.
- `MANAGER`에게는 비활성화 버튼만 보인다.
- 완전 삭제 버튼 클릭 시 확인 모달이 열린다.
- 삭제 성공 후 목록이 갱신되고 성공 메시지가 보인다.
- 삭제 차단 응답은 모달 또는 화면 메시지로 표시된다.
- 작성자 본인에게 `DRAFT`, `CANCELED` 신청서 삭제 버튼이 보인다.
- 결재중/승인/반려 신청서에는 삭제 버튼이 보이지 않는다.

### E2E

스테이징 체험 전 최소 smoke로 다음 흐름을 확인한다.

- 관리자 로그인 후 테스트 기준정보 생성, 완전 삭제
- 참조 중인 기준정보 완전 삭제 차단
- 신청자 로그인 후 임시저장 신청서 생성, 삭제
- 관리자 로그인 후 취소 신청서 삭제

## 11. 완료 기준

- 설계된 hard-delete API가 구현되어 있다.
- 기존 비활성화 API와 UX가 깨지지 않는다.
- 삭제 불가 조건이 `409 Conflict`로 일관되게 반환된다.
- 신청서 삭제는 `DRAFT`, `CANCELED`로 제한된다.
- 첨부파일 실제 파일 삭제가 검증된다.
- 프론트 테스트와 백엔드 테스트가 통과한다.
- 기존 Task 15 모바일/PWA 변경과 충돌하지 않는다.
- 운영 문서에 비활성화와 완전 삭제의 차이가 반영된다.
