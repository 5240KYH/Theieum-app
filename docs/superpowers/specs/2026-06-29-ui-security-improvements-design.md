## 목표

사용자 편의성과 보안 안정성을 함께 개선한다. 신청, 결재, 관리자 작업에서 실수 가능성을 줄이고, 권한과 첨부 접근처럼 민감한 흐름은 서버 검증과 화면 피드백이 같은 방향을 보도록 정리한다.

또한 Codex가 단일 작업자가 아니라 병렬 AI 에이전트 팀처럼 움직이도록 역할, 파일 소유 범위, 보고 게이트, 검증 책임을 고정한다. 사용자는 최종 검증자로서 설계안, 작업 분배, 중간 통합, 최종 검증 결과를 보고받고 승인한다.

## 현재 관찰

- 프론트엔드는 로그인, 대시보드, 신청서 작성/상세/내 목록, 결재함, 관리자 기준정보, 관리자 캘린더, 전체 신청서, 알림 로그로 구성되어 있다.
- 최근 목록 화면 검색조건은 `frontend/src/shared/SearchConditionPanel.tsx`를 공통으로 사용하도록 정리되어 있다.
- 서버는 `/api/auth/login` 외 `/api/**`를 인증 필수로 두고, JWT 검증 후 활성 사용자만 인증한다.
- 신청서 상세와 첨부 content는 작성자, 관리자, 매니저, 읽기 가능한 결재자 중심으로 제한되어 있다.
- 권한 회귀 테스트는 `backend/src/test/java/com/theieum/approval/api/ApiAuthorizationTest.java`에 이미 축적되어 있다.
- 화면 회귀 테스트는 `frontend/src/**/*test.tsx`, 실제 흐름 검증은 `e2e/tests/**`에 있다.

## 포함 범위

- `MANAGER`가 전체 신청서에서 상세 화면으로 이동할 때 프론트 라우팅과 서버 권한이 일치하도록 한다.
- 신청서 작성 화면에서 필수 입력, 첨부 개수, 첨부 용량, 제출 전 확인을 더 명확하게 보여준다.
- 결재함에서 승인 전 핵심 정보 확인, 첨부 확인 진입, 고액 또는 민감 작업에 대한 실수 방지 피드백을 검토한다.
- 관리자 사용자/권한 관리에서 역할 변경, 비밀번호 초기화, 완전 삭제 같은 위험 작업을 더 명확히 확인하게 한다.
- 관리자 월별 첨부 ZIP 다운로드 전 범위와 민감 첨부 가능성을 사용자가 인지하도록 한다.
- 첨부 파일 저장 경로가 root 밖으로 벗어나지 않는지 서버 검증을 보강한다.
- 역할 문자열 정규화 정책을 통일한다. 기존 `MANGER` 오타 호환은 유지할지 제거할지 구현 계획 단계에서 명확히 결정한다.

## 제외 범위

- 운영용 SSO, refresh token, httpOnly cookie 전환은 이번 범위에 포함하지 않는다.
- 대규모 디자인 시스템 개편이나 전체 CSS 재작성은 하지 않는다.
- 결재 정책, 조직 구조, 결재선 계산 로직 자체는 변경하지 않는다.
- seed 데이터, Docker volume 삭제, 운영 DB 초기화는 하지 않는다.
- 사용자가 별도로 요청하기 전까지 `git add`, `git commit`, `git push`는 하지 않는다.

## 권장 접근

### 1. 우선 수정 후보

첫 번째 작업은 서버와 화면 정책이 이미 불일치하는 `MANAGER` 상세 접근 문제다. 서버는 매니저 상세 조회를 허용하지만 프론트 `/applications/:id` route가 `MANAGER`를 제외하면 전체 신청서 상세 이동 UX가 막힐 수 있다. 이 문제는 범위가 작고 검증이 명확하므로 첫 구현 후보로 적합하다.

### 2. 사용자 편의 UX

신청서 작성은 사용자가 가장 자주 실패하는 입력 지점이다. 필수값 오류를 하나의 일반 메시지로 끝내지 않고, 사용자가 바로 수정할 필드와 이유를 알 수 있게 한다. 첨부는 현재 서버 제한이 있으므로 화면에서도 파일 수와 용량 제한을 제출 전에 알려야 한다.

결재함은 승인 버튼이 업무적으로 중요한 상태 전이를 만든다. 승인 전 신청자, 사용처, 금액, 첨부 유무, 의견을 한 번 더 확인하게 하면 실수 승인 가능성을 줄일 수 있다. 반려 의견 필수 정책은 유지한다.

관리자 화면은 권한 변경과 삭제처럼 되돌리기 어려운 작업이 모여 있다. 역할 변경 전후 차이를 보여주고, 비밀번호 초기화와 완전 삭제에는 대상 사용자명 또는 확인 문구 입력을 요구하는 방향이 적합하다.

### 3. 보안 보강

첨부 파일 저장소는 DB에 저장된 `file_path`를 기준으로 읽고 삭제한다. 저장소 root 하위 검증을 추가하면 잘못된 경로 또는 오염된 row로 인한 파일 접근 위험을 줄일 수 있다.

JWT는 현재 `localStorage`에 저장된다. 이번 작업에서 저장 방식을 크게 바꾸지는 않지만, React의 기본 escaping을 유지하고 `dangerouslySetInnerHTML` 도입을 금지하며, 알림/사용처/파일명 표시가 HTML로 해석되지 않도록 설계 원칙에 명시한다.

역할 문자열은 프론트 일부와 관리자 API에서 `MANGER`를 `MANAGER`로 보정하지만, 모든 서버 권한 체크가 같은 정책을 쓰지는 않는다. 구현 계획에서 `MANGER`를 호환 alias로 유지할지, 데이터 정리 후 차단할지 선택해야 한다. 단기적으로는 기존 데이터 호환을 위해 한 곳의 helper로 정규화하는 방안이 안전하다.

## 병렬 AI 에이전트 팀 구성

### Coordinator

담당: 전체 설계, 에이전트 분배, 충돌 조정, 사용자 보고, 최종 통합.

책임:

- 사용자 승인을 받기 전 코드 구현을 시작하지 않는다.
- 에이전트별 파일 소유 범위를 지정한다.
- 같은 파일을 둘 이상이 수정해야 하면 순서를 정한다.
- 최종 보고에 변경 파일, 검증 명령, 남은 위험을 포함한다.

### UI/UX Agent

담당 범위: `frontend/src/**`, 특히 신청서 작성/상세, 결재함, 관리자 신청서, 관리자 기준정보, 모바일 레이아웃.

책임:

- 기존 React/Vite 구조와 CSS 패턴을 따른다.
- 목록 검색조건은 `SearchConditionPanel`을 우선 사용한다.
- 모바일 화면은 버튼 텍스트 잘림, 카드 겹침, 하단 내비게이션 침범을 확인한다.
- 화면 변경 시 관련 Vitest와 가능한 브라우저 확인 시나리오를 제안한다.

### Security/API Agent

담당 범위: `backend/src/main/java/**`, `backend/src/test/java/**`, 인증/권한/첨부/관리자 API.

책임:

- 서버 권한 체크를 프론트 표시 정책보다 우선한다.
- 민감정보를 로그나 보고에 노출하지 않는다.
- 권한 변경은 `ApiAuthorizationTest` 또는 관련 서비스 테스트로 고정한다.
- 첨부 경로, MIME, 파일 크기, 관리자 권한 경계를 확인한다.

### Verification Agent

담당 범위: 테스트와 수동 확인.

책임:

- `cd frontend && npm run test`
- `cd frontend && npm run build`
- `cd backend && ./gradlew test` 또는 Java 21 Docker fallback
- `docker compose config`
- `git diff --check`
- 화면 변경이 있으면 390px 모바일과 데스크톱 브라우저 확인 결과를 보고한다.

### Docs/Handoff Agent

담당 범위: `docs/superpowers/specs/**`, `docs/superpowers/plans/**`, `docs/handoffs/**`, `docs/handoff-2026-06-03.md`.

책임:

- 설계, 계획, 인수인계 문서를 한국어로 작성한다.
- 과거 상태와 최신 상태가 섞이지 않도록 날짜와 기준 커밋을 분명히 쓴다.
- 검증 명령과 실제 결과를 문서에 남긴다.
- 문서 초안은 Coordinator 검토 후 반영한다.

## 보고 게이트

### Gate 1. 설계 승인

사용자는 이 설계 문서를 검토하고 포함 범위와 제외 범위를 승인한다. 승인 전 구현은 시작하지 않는다.

### Gate 2. 작업 분배 승인

Coordinator는 구현 계획 문서에 에이전트별 파일 소유 범위와 테스트 명령을 적고 사용자에게 보고한다. 사용자가 승인하면 에이전트 작업을 시작한다.

### Gate 3. 중간 통합 보고

각 에이전트는 변경 파일, 의도, 테스트 결과, 남은 위험을 짧게 보고한다. Coordinator는 같은 파일 충돌 여부를 확인하고 통합 순서를 정한다.

### Gate 4. 검증 보고

자동 테스트, 빌드, Docker config, 브라우저 확인 결과를 한 번에 보고한다. 실패한 검증과 실행하지 못한 검증은 이유를 함께 적는다.

### Gate 5. 사용자 최종 검증

사용자는 실제 화면과 핵심 보안 시나리오를 확인한다. 완료 판정, 추가 수정, 커밋/푸시 여부는 사용자가 결정한다.

## 충돌 방지 규칙

- 기존 작업 트리 변경은 사용자 변경으로 보고 되돌리지 않는다.
- `frontend/src/app/styles.css`, `frontend/src/app/router.tsx`, `backend/src/main/java/com/theieum/approval/application/ApplicationController.java`, `backend/src/main/java/com/theieum/approval/admin/AdminController.java`는 통합 지점으로 취급한다.
- UI와 API가 같은 계약을 바꿔야 하면 API type 또는 response 변경을 먼저 고정한 뒤 UI를 맞춘다.
- Docker 검증에서 `docker compose down -v`는 사용자의 명시 승인 없이는 실행하지 않는다.
- 임시 compose project와 task별 이미지 누적은 피하고, 필요한 경우 기존 안정 이미지명 사용을 우선한다.

## 성공 기준

- 매니저, 관리자, 신청자, 결재자의 화면 접근과 서버 권한이 서로 어긋나지 않는다.
- 신청서 작성자는 제출 전 어떤 필드를 고쳐야 하는지 즉시 알 수 있다.
- 결재자는 승인 전에 핵심 신청 정보를 다시 확인할 수 있다.
- 관리자는 위험 작업의 대상과 영향을 확인한 뒤 실행한다.
- 첨부 파일 접근은 서버 root-path 경계 안에서만 동작한다.
- 관련 프론트 테스트, 백엔드 권한 테스트, 빌드, 공백 검증이 통과한다.
- 사용자가 최종 검증자로서 변경 요약과 검증 결과를 보고받는다.

## 검증 전략

- 프론트 단위 검증: 변경 화면별 targeted Vitest 후 전체 `npm run test`.
- 프론트 빌드 검증: `cd frontend && npm run build`.
- 백엔드 권한 검증: `ApiAuthorizationTest`, 첨부 저장소 테스트, 관련 서비스 테스트.
- 전체 백엔드 검증: `cd backend && ./gradlew test`.
- 문서/공백 검증: `git diff --check`.
- 화면 검증: Docker 또는 dev server에서 신청서 작성, 결재함, 관리자 사용자, 전체 신청서 상세, 월별 첨부 다운로드 흐름 확인.
