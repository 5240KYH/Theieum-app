# Task 21 체험 계정/기준정보 및 첨부파일 관리 안정화 설계

작성일: 2026-06-05, Asia/Seoul

## 목표

외부 체험을 30명 안팎까지 늘려도 계정 배정, 기준정보 확인, 첨부파일 제한, 체험 종료 정리가 흔들리지 않도록 seed 데이터와 운영 문서를 안정화한다.

## 범위

- `local` profile과 스테이징 체험용 seed 계정을 30개 이상으로 확장한다.
- 기존 핵심 계정(`admin`, `employee01`, `lead-dev`, `approver01`)과 기존 결재선 동작은 유지한다.
- 체험 전용 계정은 `trial-*` 네이밍으로 분리해 운영자가 배정하기 쉽게 한다.
- 첨부파일은 당시 기준 영수증 이미지 1개, 기본 최대 5MB 정책을 서버 설정으로 고정한다. 후속 다중 영수증 첨부 작업에서 기본 개수는 신청서당 최대 10개로 확장한다.
- 스테이징에서는 `.env.staging` 값으로 첨부 크기와 개수 제한을 조정할 수 있다.
- 체험 종료 후 DB와 첨부파일 volume 보존 또는 삭제 절차를 문서화한다.

## 비범위

- 운영용 실계정 생성 자동화는 포함하지 않는다.
- S3, MinIO 같은 외부 객체 저장소 연동은 포함하지 않는다.
- 첨부파일 여러 개 업로드 UI는 포함하지 않는다.
- Oracle VM 실제 생성과 배포는 Task 19 재개 시 처리한다.

## 설계

### 체험 계정

기본 seed는 기존 20개 계정에 체험 전용 10개 계정을 더해 총 30개 이상을 제공한다. 체험 전용 계정은 신청자, 결재자, 매니저, 관리자 역할을 나눠 둔다. 운영자는 외부 체험자에게 가능하면 개인별 계정을 배정하고, 공유 확인용 계정은 비밀번호 변경을 요청하지 않는다.

### 기준정보

조직, 직위, 기본 결재선, 개발팀 예외 결재자는 기존 구조를 유지한다. 새 체험 계정은 기존 조직과 직위에 배치하되, 기존 E2E와 핵심 회귀 테스트가 참조하는 계정 ID와 결재선은 바꾸지 않는다. seed 규모 확장으로 영향을 받는 테스트는 독립 fixture를 사용해 seed 인원 수와 분리한다.

### 첨부파일 정책

서버 기본 정책은 Task 21 당시 신청서당 영수증 이미지 1개, 파일당 5MB 이하이다. 후속 다중 영수증 첨부 작업 이후 기본 개수는 신청서당 최대 10개이다. API는 `ATTACHMENT_MAX_IMAGE_BYTES`로 크기 제한을 읽고, 서비스는 `ATTACHMENT_MAX_FILES_PER_APPLICATION`으로 신청서당 첨부 개수 제한을 읽는다. 서비스는 제한 초과 시 파일 저장 전에 차단해 DB row와 실제 파일이 어긋나지 않도록 한다.

### 운영 정리

스테이징 체험 종료 시 운영자는 피드백 분석 필요 여부에 따라 DB와 첨부 volume을 보존하거나 삭제한다. 삭제가 필요하면 `docker compose --env-file .env.staging -f docker-compose.staging.yml down -v`로 DB와 첨부 volume을 함께 제거한다. 보존이 필요하면 `stop` 후 백업 여부를 결정한다.

## 테스트

- Flyway seed 테스트가 30개 이상 계정과 `trial-*` 계정을 확인한다.
- 신청서 서비스 테스트가 신청서당 첨부 개수 제한을 확인한다.
- API 테스트가 설정 기반 첨부 크기 제한을 확인한다.
- 기존 신청, 결재, 권한 테스트가 새 seed 확장 후에도 통과해야 한다.

## 결정 사항

- 기본 첨부 개수는 Task 21 당시 현재 UI와 맞춰 1개로 유지했다. 후속 작업에서 UI와 서버 기본값을 신청서당 최대 10개로 확장한다.
- 기본 첨부 크기는 기존 프론트 문구와 맞춰 5MB로 유지한다.
- 체험 전용 계정은 운영용 계정이 아니므로 운영 배포에서는 `db/seed`를 제외한다.
