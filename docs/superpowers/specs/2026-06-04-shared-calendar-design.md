## 1. 목적

Task 17의 목적은 모든 로그인 사용자가 같은 일정을 확인할 수 있는 내부 공용 캘린더를 추가하는 것이다. 대시보드에서 가까운 일정을 바로 확인하고, 별도 캘린더 화면에서 일정을 등록, 수정, 삭제할 수 있게 만든다.

이번 작업은 Google Calendar 연동 전 단계다. 먼저 앱 내부에 안정적인 일정 도메인과 화면을 만들고, 이후 Google Calendar 동기화 또는 가져오기 기능을 붙일 수 있도록 데이터 모델과 API 경계를 분리한다.

## 2. 배경과 현재 상태

현재 앱의 대시보드는 전자결재 업무 상태를 확인하는 첫 화면이다. 사용자는 신청서, 결재함, 알림을 확인할 수 있지만, 조직 공지성 일정이나 내부 마감 일정을 함께 볼 수 있는 영역은 없다.

사용자 요청의 핵심은 다음과 같다.

- 대시보드에서 캘린더를 확인하고 싶다.
- 캘린더 일정을 관리할 수 있으면 좋겠다.
- 접속하는 모든 대상자가 같은 캘린더 일정을 확인할 수 있으면 좋겠다.
- 추후 Google Calendar 연동까지 가능하면 좋다.

따라서 Task 17은 "모든 사용자가 볼 수 있는 공용 일정"을 우선 구현한다. 개인 일정, 초대/참석자 응답, 외부 캘린더 양방향 동기화는 이번 범위에서 제외한다.

## 3. 범위

- 일정 목록 조회 API
- 월별 기간 조회 API
- 일정 생성, 수정, 삭제 API
- 대시보드 미니 캘린더와 오늘/다가오는 일정 표시
- 별도 캘린더 화면
- 모바일 친화적인 월간 캘린더와 날짜별 일정 목록
- 일정 생성/수정 모달
- 일정 삭제 확인 모달
- 백엔드 서비스/API 테스트
- 프론트 단위 테스트
- 관리자/사용자 가이드 보강

### 제외

- Google Calendar 직접 연동
- 개인 캘린더
- 참석자 초대, 참석 여부, 회의실 예약
- 반복 일정
- 일정 알림 발송
- 파일 첨부
- 음력/공휴일 자동 표시

- 휴지통 또는 삭제 복구

반복 일정과 Google Calendar 연동은 데이터 구조가 더 복잡해지므로, 내부 공용 캘린더가 실제 사용 가능한 상태가 된 뒤 별도 task로 다룬다.

## 4. 권한 정책

일정 조회는 모든 로그인 사용자에게 허용한다.

- `ADMIN`: 일정 생성, 수정, 삭제 가능
- `MANAGER`: 일정 생성, 수정, 삭제 가능
- `APPROVER`: 일정 조회만 가능
- `APPLICANT`: 일정 조회만 가능

일정은 조직 전체에 공개되는 공용 정보로 본다. 작성자 본인만 수정하는 개인형 정책은 이번 범위에 맞지 않으므로 사용하지 않는다. 대신 등록/수정 권한을 `ADMIN`과 `MANAGER`로 제한해 일정 품질과 운영 책임을 명확히 한다.

## 5. 데이터 모델

새 테이블은 `calendar_events`로 둔다.

필드:

- `id`: bigint PK
- `title`: 일정 제목, 필수
- `description`: 상세 내용, 선택
- `location`: 장소, 선택
- `start_at`: 시작 일시, 필수
- `end_at`: 종료 일시, 필수
- `all_day`: 종일 일정 여부, 필수
- `created_by_id`: 생성 사용자 FK
- `updated_by_id`: 마지막 수정 사용자 FK
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

삭제는 물리 삭제를 기본으로 한다. 일정은 결재 이력처럼 법적/업무 기록 보호가 필요한 데이터가 아니며, 사용자가 잘못 등록한 공지성 일정을 운영자가 정리할 수 있어야 한다. 다만 삭제 전 확인 모달을 두고, 삭제 권한은 `ADMIN`과 `MANAGER`로 제한한다.

기간 규칙:

- `end_at`은 `start_at`보다 늦어야 한다.
- 종일 일정은 날짜 단위로 입력하되, 백엔드에는 해당 날짜의 시작/종료 범위를 명확한 일시로 저장한다.
- 프론트 표시 기준 시간대는 `Asia/Seoul`로 둔다.

## 6. API 설계

기본 경로는 `/api/calendar/events`로 둔다.

조회:

- `GET /api/calendar/events?from=2026-06-01T00:00:00%2B09:00&to=2026-07-01T00:00:00%2B09:00`
- 로그인 사용자 전체 허용
- 지정 기간과 겹치는 일정을 시작일 오름차순으로 반환

생성:

- `POST /api/calendar/events`
- `ADMIN`, `MANAGER`만 허용

수정:

- `PUT /api/calendar/events/{id}`
- `ADMIN`, `MANAGER`만 허용

삭제:

- `DELETE /api/calendar/events/{id}`
- `ADMIN`, `MANAGER`만 허용

응답 DTO:

- `id`
- `title`
- `description`
- `location`
- `startAt`
- `endAt`
- `allDay`
- `createdBy`
- `updatedBy`
- `createdAt`
- `updatedAt`

에러 응답:

- 인증 없음: `401 Unauthorized`
- 권한 없음: `403 Forbidden`
- 대상 없음: `404 Not Found`
- 시작/종료 시간 오류: `400 Bad Request`

사용자가 그대로 이해할 수 있는 한국어 메시지를 내려준다.

## 7. 프론트 UX

### 대시보드

대시보드에는 공용 캘린더 요약을 추가한다.

- 오늘/다가오는 일정 요약
- 주단위 일정 목록
- 월단위 일정 목록
- 보기 전환 세그먼트
- "전체 캘린더" 이동 버튼

대시보드는 반복 방문이 많은 화면이므로, 캘린더 요약은 업무 카드들과 균형 있게 배치한다. 화면을 과하게 장식하지 않고, 일정을 빠르게 스캔할 수 있는 밀도와 여백을 우선한다.

### 캘린더 화면

새 라우트는 `/calendar`로 둔다.

주요 구성:

- 상단 월 이동 컨트롤
- 월/주/목록 보기 전환
- 월간/주간 캘린더 그리드
- 날짜 셀 내부 일정 칩
- 날짜별 빠른 일정 추가 버튼
- 선택 날짜 일정 목록
- 전체 월 일정 목록
- `ADMIN`/`MANAGER` 전용 일정 등록 버튼
- 일정 클릭 시 상세/수정 모달

일반 사용자는 등록/수정/삭제 버튼을 볼 수 없다. 일정 상세는 조회 전용으로 열 수 있다.

### 모바일

모바일에서는 월간 그리드와 목록을 같이 쓰되, 조작 밀도를 낮춘다.

- 하단 탭 또는 더보기 메뉴에서 캘린더 접근
- 월 이동 버튼은 터치 영역을 충분히 확보
- 날짜 셀에는 일정 칩 또는 보조 개수 표시
- 선택 날짜 아래에 일정 카드 목록 표시
- 등록/수정 모달은 작은 화면에서 세로 스크롤 가능

Task 15에서 추가한 모바일/PWA 방향과 맞춰, 캘린더도 앱처럼 한 손으로 확인할 수 있는 구조를 따른다.

## 8. 일정 입력 폼

필드:

- 제목, 필수
- 시작 날짜/시간, 필수
- 종료 날짜/시간, 필수
- 종일 일정 여부
- 장소, 선택
- 상세 내용, 선택

입력 흐름:

- 캘린더 앱처럼 제목 입력을 가장 먼저 배치한다.
- 시작/종료 시간과 종일 여부는 같은 흐름에서 빠르게 조정한다.
- 월간 캘린더 날짜 셀에서 추가 버튼을 누르면 해당 날짜의 기본 09:00-10:00 일정으로 열린다.
- 일반 사용자가 일정 칩을 누르면 조회 전용 상세 모달로 열린다.

검증:

- 제목은 공백만 입력할 수 없다.
- 종료는 시작보다 늦어야 한다.
- 종일 일정이면 시간 입력을 숨기거나 비활성화한다.
- 저장 중에는 중복 제출을 막는다.
- 저장 실패 시 모달 내부에 서버 메시지를 표시한다.

삭제:

- 일정 상세/수정 모달에서 삭제 버튼 제공
- 삭제 전 "복구할 수 없습니다." 확인 모달 표시
- 삭제 성공 후 현재 월 데이터를 다시 조회

## 9. 데이터 흐름

1. 사용자가 대시보드 또는 캘린더 화면에 진입한다.
2. 프론트가 현재 월 시작/종료 범위로 일정 조회 API를 호출한다.
3. 백엔드가 기간과 겹치는 일정을 조회한다.
4. 프론트가 월간 날짜별로 일정을 그룹화한다.
5. 사용자가 날짜를 선택하면 해당 날짜 일정을 강조 표시한다.
6. `ADMIN` 또는 `MANAGER`가 일정을 등록/수정/삭제한다.
7. 성공 후 프론트가 현재 월 일정을 다시 조회한다.

대시보드는 같은 API를 사용하되, 오늘/다가오는 일정 표시를 위해 현재 날짜 기준으로 클라이언트에서 필터링한다.

## 10. Google Calendar 연동을 위한 경계

Task 17에서는 외부 연동 코드를 넣지 않는다. 대신 다음 경계를 남긴다.

- 내부 일정 CRUD는 `CalendarEventService`에서 담당한다.
- 외부 연동은 후속 task에서 `CalendarSyncService` 같은 별도 서비스로 붙인다.
- `calendar_events`에 당장 Google event id를 넣지 않는다.
- Google Calendar 연동이 필요해지는 시점에 별도 매핑 테이블을 추가한다.

후속 Google 연동 범위 후보:

- Google Calendar 일정을 내부 캘린더로 읽어오기
- 내부 일정 생성 시 Google Calendar에도 생성하기
- 특정 공유 캘린더와 단방향 또는 양방향 동기화하기

초기에는 "Google Calendar 가져오기" 단방향 연동이 가장 안전하다. 양방향 동기화는 충돌 처리와 삭제 정책이 필요하므로 별도 설계가 필요하다.

## 11. 테스트 전략

백엔드:

- 로그인 사용자는 일정 목록을 조회할 수 있다.
- `ADMIN`은 일정을 생성, 수정, 삭제할 수 있다.
- `MANAGER`는 일정을 생성, 수정, 삭제할 수 있다.
- `APPLICANT`와 `APPROVER`는 생성, 수정, 삭제할 수 없다.
- 종료가 시작보다 빠르거나 같은 일정은 저장할 수 없다.
- 기간 조회는 범위와 겹치는 일정만 반환한다.

프론트:

- 대시보드가 일정 요약을 표시한다.
- 일정이 있는 날짜가 월간 캘린더에 표시된다.
- 일반 사용자는 일정 관리 버튼을 볼 수 없다.
- `ADMIN` 또는 `MANAGER`는 일정 등록 버튼을 볼 수 있다.
- 일정 생성/수정/삭제 성공 후 목록이 갱신된다.
- 모바일 폭에서 캘린더 화면의 주요 버튼과 목록이 겹치지 않는다.
