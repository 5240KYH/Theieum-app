# 영수증 첨부 전자결재 앱 설계서

## 1. 목적

20명 내외 인원이 사용할 수 있는 영수증 첨부 전자결재 앱을 만든다.

첫 버전은 조직/직위/대상자 등록, 결재선 등록, 영수증 첨부 신청서 작성, 결재 요청, 승인/반려, 최종 완료 알림, 전체 관리 화면을 제공한다. 이후 카카오 알림, 이메일 알림, S3 계열 파일 저장소, 복잡한 결재 정책을 단계적으로 확장할 수 있도록 기본 구조를 분리한다.

## 2. MVP 범위

### 포함

- 관리자 등록 계정 기반 아이디/비밀번호 로그인
- 사용자, 조직, 직위 등록 및 관리
- 상위 조직을 갖는 조직 트리 관리
- 전자결재 유형별 결재선 등록
- 결재 단계별 직접 대상자 지정
- 결재 단계별 조직+직위 기준 결재자 산정
- 조직별 예외 결재자 목록 설정
- 영수증 첨부 신청서 작성, 임시저장, 제출, 취소
- 이미지 첨부, 썸네일, 미리보기, 삭제
- 결재자 승인/반려
- 관리자 예외 결재
- 신청서 상태 및 결재 이력 조회
- 인앱 알림함
- 전체 신청서 관리 화면
- 알림 로그 관리 화면
- Docker Compose 기반 로컬/운영 배포 하네스
- 테스트용 20명 규모 seed 데이터

### 제외

- 대리결재
- 병렬결재
- 전결
- 금액별 조건 결재선
- 진행 중 결재자 변경
- OCR 자동 인식
- 회계 시스템 연동
- 전자서명/인증서
- 모바일 앱 별도 개발
- 복잡한 양식 빌더

단, 제외 항목 중 카카오 알림과 외부 파일 저장소는 구조상 확장 가능하도록 인터페이스와 데이터 모델을 준비한다.

## 3. 기술 구성

### Backend

- Java 21
- Spring Boot
- Spring Security
- Spring Data JPA
- PostgreSQL
- Flyway
- JUnit 5
- Testcontainers

### Frontend

- React
- Vite
- TypeScript
- TanStack Query
- React Router
- 표, 폼, 모달, 첨부 미리보기에 적합한 컴포넌트 구조

### 배포

- Docker Compose
- backend 컨테이너
- frontend Vite 빌드 결과를 nginx 컨테이너에서 서빙
- PostgreSQL 컨테이너
- nginx reverse proxy
- 파일 저장용 Docker volume

MVP에서는 로컬 볼륨에 첨부파일을 저장한다. 파일 저장 기능은 `FileStorage` 인터페이스 뒤로 숨겨 이후 MinIO 또는 S3 호환 저장소로 교체할 수 있게 한다.

## 4. 시스템 구조

```text
React/Vite Frontend
  -> Spring Boot REST API
    -> PostgreSQL
    -> FileStorage
    -> NotificationEvent
      -> InAppNotificationSender
      -> EmailNotificationSender, 향후 추가
      -> KakaoNotificationSender, 향후 추가
```

백엔드는 인증, 사용자/조직/직위, 결재선, 신청서, 파일, 알림 이벤트를 책임진다. 프론트엔드는 홈, 신청서 작성, 내 신청서, 결재함, 관리 화면을 제공한다.

알림은 결재 도메인 로직과 직접 결합하지 않는다. 결재 요청, 승인, 반려, 최종 완료 같은 업무 이벤트가 발생하면 먼저 `notification_events`에 저장하고, 채널별 발송기가 이를 처리한다.

## 8. 신청서 상태

```text
DRAFT -> SUBMITTED -> IN_APPROVAL -> APPROVED
                         -> REJECTED
DRAFT -> CANCELED
```

- `DRAFT`: 임시저장
- `SUBMITTED`: 제출 접수
- `IN_APPROVAL`: 결재 진행 중
- `APPROVED`: 최종 승인 완료
- `REJECTED`: 반려
- `CANCELED`: 기안자 취소

## 9. 결재 단계 상태

```text
PENDING -> APPROVED
PENDING -> REJECTED
PENDING -> ADMIN_APPROVED
```

- `PENDING`: 결재 대기
- `APPROVED`: 결재자 승인
- `REJECTED`: 결재자 반려
- `ADMIN_APPROVED`: 관리자 예외 승인

## 10. 결재선 산정 규칙

신청서 제출 시 결재선은 스냅샷으로 고정한다. 제출 이후 조직, 직위, 결재선 설정이 변경되어도 진행 중 신청서의 결재자는 변경하지 않는다.

산정 순서는 다음과 같다.

1. 신청서의 전자결재 유형을 확인한다.
2. 신청자 조직에 조직별 예외 결재자 목록이 있으면 이를 우선 적용한다.
3. 예외가 없으면 전자결재 유형의 기본 결재선을 조회한다.
4. 결재 단계별로 실제 결재자를 산정한다.
5. 산정된 결재자를 `application_approval_steps`에 저장한다.
6. 첫 번째 결재자에게 인앱 알림을 생성한다.

동일 조직 안에 결재자가 여러 명이면 직위 순서를 기본으로 한다. 직접 대상자 지정 방식에서는 결재선 등록 화면에서 입력한 순서를 우선한다.

## 11. 관리자 예외 결재

대리결재는 MVP에서 제공하지 않는다. 대신 특정 결재자가 결재할 수 없는 경우 관리자가 현재 대기 단계에 대해 예외 결재할 수 있다.

정책은 다음과 같다.

- 관리자만 예외 결재 가능
- 현재 `PENDING` 단계만 예외 결재 가능
- 예외 결재 시 사유 입력 필수
- 원래 결재자와 실제 처리자를 모두 저장
- 신청서 상세와 관리자 화면에 `관리자 예외 처리` 표시
- 최종 완료 또는 반려 이후에는 예외 결재 불가

관리자 예외 결재는 운영 장애 해소용 기능이며, 권한 위임이나 대리결재 관리 기능으로 확장하지 않는다.

## 12. 알림 설계

MVP 알림은 인앱 알림으로 구현한다.

알림 발생 시점은 다음과 같다.

- 기안자가 결재 요청을 제출했을 때 첫 결재자에게 알림
- 결재자가 승인했을 때 다음 결재자에게 알림
- 최종 결재가 완료되었을 때 기안자에게 알림
- 반려되었을 때 기안자에게 알림
- 관리자가 예외 결재했을 때 기안자와 다음 결재자에게 필요한 알림

카카오 알림은 MVP 구현 대상은 아니지만, 다음 구조를 준비한다.

```text
NotificationSender
  - InAppNotificationSender
  - EmailNotificationSender
  - KakaoNotificationSender
```

`notification_events`에는 카카오 연동에 필요한 `template_code`, `external_message_id`, `failed_reason`을 저장할 수 있게 한다. 실제 카카오 연동 시에는 결재 로직을 수정하지 않고 발송기와 채널 설정만 추가한다.

## 13. 보안과 권한

- 비밀번호는 해시로 저장한다.
- 일반 사용자는 본인 신청서만 조회한다.
- 결재자는 본인에게 배정된 결재 건만 처리한다.
- 관리자는 전체 신청서와 설정 데이터를 조회/수정할 수 있다.
- 첨부파일은 권한 확인 후 다운로드 또는 미리보기를 허용한다.
- 관리자 예외 결재는 반드시 감사 이력을 남긴다.

## 14. 하네스 엔지니어링

품질 목표는 단순 실행이 아니라 업무 흐름의 반복 검증이다.

### 로컬 실행 하네스

- `docker compose up`으로 전체 시스템 실행
- PostgreSQL 자동 기동
- Flyway migration 자동 적용
- 20명 규모 seed 데이터 자동 생성
- 관리자, 기안자, 결재자 테스트 계정 제공
- 첨부파일 저장 볼륨 구성

### Backend 테스트

- 결재선 산정 단위 테스트
- 신청서 제출 통합 테스트
- 승인/반려 통합 테스트
- 관리자 예외 결재 테스트
- 알림 이벤트 생성 테스트
- 첨부파일 업로드 테스트
- 권한 검증 테스트

### Frontend 테스트

- 신청서 필수값 검증
- 첨부 이미지 선택/삭제/미리보기
- 결재함 상태 표시
- 관리자 화면 필터

### E2E 테스트

Playwright로 다음 흐름을 검증한다.

1. 관리자 로그인
2. seed 사용자/조직/직위 확인
3. 기안자 로그인
4. 신청서 작성
5. 영수증 이미지 첨부
6. 결재 요청 제출
7. 결재자 로그인
8. 알림 확인
9. 신청서 승인
10. 마지막 결재 완료
11. 기안자 최종 승인 알림 확인
12. 관리자 전체 현황 확인
13. 관리자 예외 결재 시나리오 확인

## 16. 승인된 주요 결정

- 기술스택은 Spring Boot REST API, React/Vite, PostgreSQL, Docker Compose로 진행한다.
- 로그인은 관리자 등록 계정과 아이디/비밀번호 방식으로 시작한다.
- 신청서 필수 항목은 신청일자, 신청자, 영수증 일자, 사용처, 금액, 신청 내용, 영수증 이미지 첨부로 한다.
- 결재선은 제출 시점 스냅샷으로 고정한다.
- MVP 결재 단계 유형은 직접 대상자 지정과 조직+직위 기준만 지원한다.
- 대리결재는 제외한다.
- 대신 관리자 예외 결재를 제공한다.
- MVP 알림은 인앱 알림으로 구현한다.
- 카카오 알림은 초기부터 확장 가능한 채널 구조와 컬럼을 준비한다.
