## 4. Task 11 구현 시 주의점

Docker Compose는 MVP 확인을 위해 `SPRING_PROFILES_ACTIVE=local`을 사용합니다. 이 profile은 seed 데이터를 함께 적용합니다.

Compose의 현재 방향:

- PostgreSQL: `approval` DB
- Backend: `backend:8080`, 업로드 볼륨 `/app/uploads`
- Frontend: nginx, `localhost:3000`
- nginx `/api/*` 프록시: `http://backend:8080/api/*`

운영 배포 전에는 반드시 교체해야 합니다.

- `JWT_SECRET`
- DB 비밀번호
- 파일 저장 경로
- seed/local profile 사용 여부

## 7. 열린 이슈와 리스크

- `.idea/`는 IDE 메타데이터라 `.gitignore`에 추가했으며 커밋 대상이 아닙니다.
- 기본 프로젝트의 기존 `theieum_postgres-data` 볼륨에는 이전 Flyway V2 checksum이 남아 있어 그대로 기동하면 백엔드가 실패합니다. 기존 볼륨은 보존했고, 깨끗한 검증용 프로젝트에서는 Task 11 하네스가 정상 기동했습니다.
- 프론트 관리 CRUD는 현재 목록/조회 중심입니다. 백엔드가 `GET`/`POST`만 제공하므로 본격 수정/삭제 기능은 추후 API 확장이 필요합니다.
- 결재선 생성에는 `approvalTypeId`가 필요하지만 approval type 목록 API가 없습니다.
- 관리자 알림 로그 API는 `title`, `body`, `createdAt`, `recipientName` 등을 제공하지 않습니다. 현재 화면은 제공 필드 중심으로 표시합니다.
- 개인 알림함은 pagination/unread-count API가 없어 전체 목록 조회 후 프론트에서 미확인 수를 계산합니다.
